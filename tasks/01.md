# Painel Administrativo — Customização do Sanity Studio

**Objetivo:** Transformar o Sanity Studio (`/admin`) em um painel administrativo simplificado para um cliente não técnico, removendo funcionalidades desnecessárias e limitando permissões pela configuração do Studio.

---

## Arquivos afetados

| Arquivo | O que muda |
|---|---|
| `sanity.config.ts` | Adicionar locale pt-BR, remover `visionTool`, configurar `document.actions` e `document.newDocumentOptions` |
| `src/sanity/structure.ts` | Menu lateral customizado (filtrar schemas, definir ordem e ícones) |

---

## 1. Interface em português (pt-BR)

Instalar o pacote oficial de locale:

```bash
npm install @sanity/locale-pt-br
```

Adicionar `ptBRLocale()` ao array `plugins` em `sanity.config.ts`:

```ts
import { ptBRLocale } from '@sanity/locale-pt-br'

export default defineConfig({
  // ...
  plugins: [
    ptBRLocale(),
    structureTool({ structure }),
  ],
})
```

Isso traduz automaticamente menus, botões, ações e textos internos do Studio.

---

## 2. Remover o visionTool

O `visionTool` é um playground GROQ para desenvolvedores. Removê-lo do array `plugins` em `sanity.config.ts`:

```diff
  plugins: [
    ptBRLocale(),
    structureTool({ structure }),
-   visionTool({ defaultApiVersion: apiVersion }),
  ],
```

---

## 3. Permissões por schema

Cada schema deve ter visibilidade e ações bem definidas:

| Schema | Visível no menu? | Ações permitidas |
|---|---|---|
| `suit` | Sim | Todas (CRUD completo) |
| `order` | Sim | Somente leitura (sem create, sem delete, sem duplicate) |
| `review` | Sim | Somente update (sem create, sem delete, sem duplicate) |
| `schedule` | Sim | Somente update (sem create, sem delete, sem duplicate) |
| `about` | Não | Oculto do menu |
| `customSuit` | Não | Oculto do menu |

---

## 4. Menu lateral (Structure Builder)

Customizar `src/sanity/structure.ts` para exibir apenas os schemas permitidos, na ordem desejada:

```
Painel Admin
  ├── Trajes (suit)           — lista de documentos
  ├── Encomendas (order)      — lista de documentos
  ├── Avaliações (review)     — lista de documentos
  └── Agenda (schedule)       — singleton (item único, sem lista)
```

Os schemas `about` e `customSuit` continuam existindo no código (o frontend depende deles), mas **não aparecem no menu lateral**.

Usar `S.documentTypeListItem()` para os tipos normais e `S.listItem()` + `S.editor()` para o singleton `schedule`.

---

## 5. Restringir ações de documento (`document.actions`)

Em `sanity.config.ts`, adicionar a propriedade `document.actions` para filtrar ações por tipo de schema.

Os nomes exatos das actions do Sanity são: `'publish'`, `'unpublish'`, `'delete'`, `'duplicate'`, `'discardChanges'`, `'restore'`.

Para `order`, `review` e `schedule`, manter apenas:
- `'publish'`
- `'unpublish'`
- `'discardChanges'`
- `'restore'`

Remover:
- `'delete'`
- `'duplicate'`

```ts
document: {
  actions: (prev, { schemaType }) => {
    if (['order', 'review', 'schedule'].includes(schemaType)) {
      return prev.filter(
        (action) => action.action !== 'delete' && action.action !== 'duplicate'
      )
    }
    return prev
  },
}
```

---

## 6. Bloquear criação de documentos (`document.newDocumentOptions`)

Complementa o passo anterior. Impedir que `order`, `review` e `schedule` apareçam no botão global "Criar novo documento":

```ts
document: {
  newDocumentOptions: (prev) => {
    const blocked = ['order', 'review', 'schedule', 'about', 'customSuit']
    return prev.filter((opt) => !blocked.includes(opt.templateId))
  },
}
```

---

## Resultado esperado

Um Sanity Studio em `/admin` onde o cliente consegue:

- Ver e gerenciar **trajes** (criar, editar, deletar)
- Ver **encomendas** recebidas (sem criar ou deletar)
- Editar **avaliações** existentes (sem criar ou deletar)
- Editar a **disponibilidade da agenda** (sem criar ou deletar)

Sem acesso a schemas internos (`about`, `customSuit`), sem ferramentas técnicas (`visionTool`), e com toda a interface em português.
