**Objetivo:** Garantir que o schema `schedule` funcione como um singleton verdadeiro — exatamente um documento com ID fixo, impossível de duplicar ou deletar, criado por padrão com agenda aberta (`isOpen: true`).

| `src/sanity/schemaTypes/scheduleType.ts` | Nenhuma mudança (já está correto) |

## Contexto — O que já está implementado (task 01)

As seguintes proteções já foram aplicadas e **não devem ser alteradas**:

- **Structure Builder** (`src/sanity/structure.ts`): renderiza `schedule` como editor único com `S.document().schemaType('schedule').documentId('schedule')`, sem lista de documentos.
- **`document.actions`** (`sanity.config.ts`): remove as ações `delete` e `duplicate` para o tipo `schedule`.
- **`document.newDocumentOptions`** (`sanity.config.ts`): bloqueia `schedule` do botão global "Criar novo documento".
- **Schema** (`src/sanity/schemaTypes/scheduleType.ts`): define `initialValue: true` para `isOpen` e validação condicional de `reopenDate`.

---

## 1. Fixar o ID do singleton via `schema.templates`

O Structure Builder já aponta para `documentId: 'schedule'`, mas o template padrão do Sanity não garante que o documento seja criado com esse ID. Configurar `schema.templates` em `sanity.config.ts` para forçar o `_id`:

```ts
schema: {
  ...schema,
  templates: (templates) =>
    templates.map((t) =>
      t.schemaType === 'schedule'
        ? { ...t, value: { _id: 'schedule', _type: 'schedule', isOpen: true } }
        : t,
    ),
},
```

Isso garante que, ao abrir o editor singleton pela primeira vez, o documento já venha com:
- `_id: 'schedule'` (ID fixo, impossível criar duplicatas)
- `_type: 'schedule'`
- `isOpen: true` (agenda aberta por padrão)

---

## 2. Resiliência no frontend (já implementado)

A query GROQ usada pelo frontend (`REOPEN_SCHEDULE_DATE` em `src/sanity/lib/queries.ts`) usa `[0]` e os consumers já tratam `null` com optional chaining:

- `src/components/landing-page/Hero.tsx`: `schedule?.isOpen` — se `null`, não renderiza o badge de agenda fechada (comportamento correto: assume aberta).
- `src/app/(frontend)/trajes/[slug]/page.tsx`: mesmo padrão — se `null`, renderiza `SuitDetails` sem `reopenScheduleDate`.

**Nenhuma mudança necessária no frontend.** O fallback já é seguro.

## Resultado esperado

- Exatamente **1 documento** `schedule` no banco, com ID fixo `'schedule'`
- Impossível criar duplicatas (bloqueado por `newDocumentOptions`, `document.actions` e ID fixo no template)
- Impossível deletar (bloqueado por `document.actions`)
- Estado padrão ao criar: **agenda aberta** (`isOpen: true`)
- Frontend resiliente se o documento ainda não existir (trata `null` como agenda aberta)
